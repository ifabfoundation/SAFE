\section{Sezione 1 -- Indagine su architetture disponibili}

L’obiettivo del progetto è quello di sviluppare un modello AI che sia in grado di predire in anticipo un’anomalia di una certa macchina.

Innanzitutto sono disponibili i dati campionati da essa con il seguente intervallo temporale:

\begin{center}
    24 ott 2024 11:18:23 -- 20 gen 2025 16:00:15
\end{center}


Campionato a 1Hz, si hanno 88g 4h 41m 52s. Sono quindi presenti 21 valori che cambiano nel tempo, di cui uno è il timestamp.

In letteratura vi è una serie di architetture e modelli preconfezionati che possano aiutarci a raggiungere il nostro scopo, ma prima conviene analizzarne i componenti alla base per capire bene dove concentrarsi nello sviluppo.

\subsection{Caso 1 -- CNN-1D}

\subsubsection{Introduzione}

Per iniziare, si è scelto di indagare l’effettiva utilità delle CNN nel nostro caso, vista la presenza abbondante in letteratura di progetti simili che integrano in toto o in parte le reti neurali convoluzionali. 
Avendo una serie multivariata, e non essendo quindi in presenza di immagini, risulta ovvia l’eventuale applicazione di CNN monodimensionale (da qui 1-D) anziché le classiche a più dimensioni. In breve, la rete in esame funziona come in seguito:


\begin{itemize}
    \item Utilizza una finestra* lunga 256 campioni, ognuno dei quali corrisponde a una variabile nel tempo. In particolare, questa finestra contiene una di queste variabili campionata ogni secondo fino quindi ad arrivare a 256s (4m 16s). Considerando che le variabili sono 20, è logico dedurre che il contenuto della finestra è di 256x20 (tensore di input).
    \item L’intervallo temporale di circa 4m ci assicura la copertura di un intero ciclo macchina, e aggiungendo un overlap del 50\% (sovrapposizione di metà finestra ad ogni iterazione) siamo sicuri di non perdere eventuali informazioni lungo i bordi.
    \item Parlando di memoria, si noti che ogni valore è float32 in allenamento, mentre a miniaturizzazione avvenuta sarà int8. Inoltre abbiamo 7 620 113 secondi di dati che, considerando l’overlap (nuova finestra ogni 128s) ci indica 59530 finestre da 20Kb circa ogni epoca.
    \item Il tipico schema di una CNN-1D è quello che permette una facile compressione della finestra in un bottleneck (encoder) e cercando di ricostruirla fedelmente (decoder). In inferenza, se questo tentativo fallisce, si ha la cosiddetta anomalia.
\end{itemize}

\subsubsection{Training}

Vengono inserite 64 finestre alla volta (regola TensorFlow) in pasto ai Layer. Vi sarà una ‘lente’ che scorre 3s (campioni) sul blocco da 256s facendo operazioni sulle loro 20 variabili. Terminate tutte le finestre in ingresso si continua con le successive e, su NVIDIA L40s disponibili presso IFAB, è questione di pochi minuti prima di terminare. Si considera solitamente un 90\% training e 10\% validation in Anomaly Detection.

    \item Metadati: 1,5kB
    \item Firmware-Kernel: 5kB
\end{itemize}
Con somma pari a 10k.

\begin{itemize}

\subsubsection{Problemi}

Il primo problema riguarda l’impostazione automatica dei frameworks di CNNs, ovvero la doppia sommatoria che considera sia la variazione dal normale di ogni singolo campione (pesato sul totale) sia quella nel tempo. Di conseguenza, se un solo valore cambia rispetto agli altri, il peso di quest’ultima variazione viene \textbf{diluito}, soprattutto nei casi di spike (picchi di pochi secondi). A noi serve poter individuare anche un singolo valore anomalo o, in ogni caso, considerare gruppi di valori tra loro correlati.

Un secondo problema più rilevante è dovuto all’architettura stessa delle reti neurali convoluzionali. Esse hanno una finestra che rileva, certamente, un intero ciclo macchina. Nel caso però di una o piu variabili che vanno alla deriva durante la giornata (lentamente, ma in modo costante) la CNN per sua natura non è in grado di leggere questo ‘drift’. È progettata solo per picchi. Conseguentemente, la rete si comporterebbe quasi(**) come un if-else e, solo quando la variabile esce completamente dalla soglia consentita, è in grado di riconoscerla come anomalia. Ma è troppo tardi.

Consideriamo dunque questo semplice esempio:
Un sensore viene considerato in \textbf{normali condizioni} operative se rileva temperature intorno ai 42°C. La soglia, fissata a 50°, indica la presenza di un problema. Ipotizzando un aumento di temperatura ogni 30min (ben fuori dalla finestra+overlap) si arriverebbe all’allarme solo al raggiungimento dei 50°C. 

Infine, l’architettura deve poter essere resistente ai glitch: vi sono casi in cui un picco deve non essere segnalato in quanto potrebbe non trattarsi di un’anomalia (con le dovute precauzioni).
Si vede dunque necessario poter implementare una rete in grado di memorizzare lo stato più a lungo termine.

\subsubsection{Note}
*Viene chiamata finestra nonostante sia a tutti gli effetti un vettore di 20 variabili anziché una matrice

**Dico ‘quasi’ poiché comunque uno spike di qualche secondo verrebbe segnalato come da definizione.

\subsection{Caso 2 -- LSTM}

\subsubsection{Introduzione}

Una rete LSTM (tipicamente nella sua versione più diffusa in anomaly detection, LSTM32Byte) funziona nel modo seguente:
\begin{itemize}
    \item Possiede uno stato di 64B (32 + 32 hidden) che riassume gli ‘avvenimenti’ precedenti
    \item Prende in ingresso il primo ‘dato’ e viene scritto in quei 64B
    \item Il dato successivo, diverso o uguale, arriva in ingresso. In quei 64B verrà annotato che per due iterazioni è arrivato in ingresso un dato uguale o diverso.
\end{itemize}
Così via, per le successive iterazioni. Chiaro dunque che l’informazione contenuta in quei 64B è un riassunto di quello che è successo e non una sovrascrittura/variazione rispetto alla precedente, si può procedere a fare alcune valutazioni.

\subsubsection{Considerazioni e problemi}

In primis, l’LSTM32 è in grado di trovare derive di componenti anche per 1 o 2 giorni (più componenti subiscono drift, e prima si satura il buffer di stato). Inoltre, questa rete è in grado di ricordare per molto tempo eventuali eventi successi, anche di pochi secondi. Questo significa che un eventuale spike può protrarsi nel buffer per qualche ora e, nel caso riavvenga per colpa di un glitch, verrebbe segnalato come anomalia.

Occorre quindi ignorare questi brevi picchi a seconda della situazione. Questo è sì possibile in LSTM, ma complicato in sviluppo poiché questa rete non nasce nativamente per trattare queste situazioni. Di conseguenza, tarare ad hoc i parametri può risultare dispendioso in termini di tempo per trovare il set adatto.

La soluzione al nostro caso di studio converge verso l’utilizzo combinato di queste tecnologie contrapposte che, se suddivise nei loro compiti, possono adattarsi a una molteplicità di scenari, tra cui il nostro.

\subsection{Caso 3 -- LSTM + CNN-1D}

\subsubsection{Introduzione}

Riconosce inoltre facilmente gli spike e, se generati da certe variabili o gruppi di queste, può decidere di ignorarli. Bisogna ora considerare quante risorse in piu si renderanno necessarie. \section{Sezione 2 -- Ipotesi sui dati e raggruppamenti}

Il nostro progetto prevede un dataset con 20 variabili di input ed essendo derivanti da una stessa macchina. È ragionevole pensare che vi siano una serie di \textbf{correlazioni} tra essi e che, in allenamento/inferenza, queste somiglianze debbano essere prese in seria considerazione. Si vuole infatti evitare il più possibile l’incontro di falsi allarmi e, soprattutto, il mancato riconoscimento di pattern pericolosi. Il raggruppamento è possibile tramite una regola di Tensorflow

\subsection{Raggruppamenti ipotetici}

\subsubsection{Gruppo A [Processo]}

m1\_p4\_current (I assorbimento),\\
system1\_p0\_payload (Portata)\\
system1\_p1\_temp (Temp uscita)\\
m1\_p4\_status (ON/OFF)

Queste variabili rappresentano probabilmente le condizioni operative del carico termico e meccanico, in altre parole lo stress della macchina. 

Considerando macchine di caratteristiche simili \textit{(quindi senza aver ancora guardato i dati precisamente)} se la temperatura sale è spesso dovuto a un incremento di corrente. Se invece la temperatura scende, può essere colpa della diluizione di carico (Payload è salito). Da notare che la temperatura solitamente reagisce dopo 2/3 finestre (10/15 min). Se invece la temperatura non segue l’andamento di I o di P, può essere indice di anomalia.

Infine, un elemento da considerare è lo status: variazioni critiche di corrente, payload e temperature in fase di spegnimento sono da considerarsi regolari.

\subsubsection{Gruppo B [Indicatori meccanici]}

temp\\
acc\_rms\\
kurt\\
cfa (crest factor)\\
skew\\
max / min,\\
Std

Tramite questi indicatori è possibile individuare \textit{pattern fisici}. Un esempio può essere il gioco meccanico (difetto): Se l’accelerometro RMS e la STD aumentano con max e min che hanno dei picchi, vi è rispettivamente molta energia vibratoria, grandi oscillazioni e urti lungo (ad esempio) un bordo macchina.

\subsubsection{Gruppo C [Incroci]}

Più \textit{difficili} da individuare, serve una conoscenza approfondita sui dati. Un ipotetico caso è l’overload (RMS alto insieme alla corrente).

La board possiede un microcontrollore principale con CPU, RAM e memoria FLASH. Inoltre su di esso sono disponibili vari bus, tra cui
\begin{itemize}
    \item $I^2C$: Bus a 2 fili (SCL clock, SDA dati) su cui piu sensori possono essere collegati e ognuno ha un indirizzo. Es. la CPU chiede a sensore 0x68 6 byte a partire da 0x00 $\rightarrow$ vengono passati in SDA a ritmo SCL.
    \item UART: due fili molto più semplici (TX invio, RX ricezione), usato per log/test tra PC e Board in debug. La CPU bufferizza in RAM interna ogni dato che arriva su RX. Naturalmente asincrono, può anche essere impostato sincrono (USART).
\end{itemize}
Inoltre va ricordata l’interfaccia \textbf{NVIC} all’interno della CPU che gestisce gli interrupts in arrivo (ad esempio da GPIO) e delega le callback specifiche per la sua gestione.

Oltre al chip primario, è presente un secondario \textbf{ST-LINK} più piccolo, vicino al connettore micro-USB, che funge da ponte con il PC e la Board. Possiede un convertire chiamato Virtual COM per rendere possibile la comunicazione USB – USART e quindi fondamentale per connettere PC e scheda. Questo chip viene utilizzato per programmare il firmware dentro al primario e per i vari debug (pause, breakpoints, lookup variabili in CubeIDE). Esempio di funzionamento:
\begin{itemize}
    \item Programmazione: quando voglio caricare il file compilato da PC sul chip F411, esso viene prima inviato a ST-LINK che tramite un’interfaccia seriale apposita SWD (Non quella USART) scrive quei byte in FLASH del microcontrollore principale tramite due pin appositi. Si occuperà quest’ultimo dell’esecuzione.
    \item Debug: la stessa interfaccia SWD viene utilizzata per fermare l’esecuzione, mettere breakpoints o leggere registri/RAM $\rightarrow$ pieno controllo
    \item Log: tramite il bridge Virtual COM, come detto prima, permette la comunicazione tramite TX/RX. In particolare, ogni printf() sul codice viene scritta su TX e ricevuta quindi dal PC tramite COM.
\end{itemize}

\subsubsection{Qualche prova}
L’interfaccia iniziale è quella del file ‘\textit{progettista}’ (.ioc). La prima azione da svolgere è quella di attivare i bus sopra menzionati: Connectivity $\rightarrow$ I2C1 $\rightarrow$ $I^2C$ 

A questo punto si illumineranno i pin della CPU di verde (possono essere cambiati volendo). In ogni caso dalla schermata in basso dei ‘Parameter Settings’ conviene attivare la Fast Mode in Master Features. 

Ora invece attiveremo \textbf{UART} (di default solitamente) lasciando quindi async al posto di sync e lasciando la configurazione 115200 bit/s 8 bit/words 1 Stop bit (ogni word finisce con un bit a 1). Questo è lo standard di qualsiasi terminale seriale richiesto di default per una comunicazione con PC. 

Non ci resta che \textit{scandire il tempo} di campionamento pari a 1Hz: Timers $\rightarrow$ TIM2 $\rightarrow$ ‘Clock Source’ = Internal Clock. Ora è necessario aprire il tab Clock configuration, che esporrà tutti i clock di ogni parte della board (es. Ethernet, Cortex, Periferiche…). Quello che ci interessa è APB1 e APB1x2, le strade più lente per le periferiche, che presentano prescaler per dividere il clock (depotenziarlo). Per questo motivo, bisogna annotarsi i due valore di APB1, tornare su Pinout configuration e configurare Prescaler-CounterPeriod seguento la \textit{formula}: 

\begin{equation}
f_{\text{update}} = \frac{f_{\mathrm{TIMxCLK}}}{(PSC + 1)\,(ARR + 1)} \, .
\end{equation}

Dal momento che il clock su APB1 vale 42Mhz, metteremo prescaler pari a 8399 e Counter Period 9999 (entrambi con il -1 dovuto al +1 della formula. Useremo dunque TIM2 come timer per il nostro modello. Spuntare inoltre ‘enabled’ in NVIC Settings per consentire al timer di mandare un \textbf{interrupt} ad ogni Hz alla CPU via NVIC.

\subsubsection{Organizzazione del file main.c }

Innanzitutto si ha una suddivisione del file in 3 parti principali:
\begin{itemize}
    \item Operazioni preliminari: includes, static functions e variabili globali.
    \item Main: Inizializza le principali funzionalità, tra cui HAL, Clock e nel nostro caso deve avviare il Timer (pronto per interrupt). Infine, prosegue con un while(1) loop che include la logica da seguire durante il lavoro.
    \item Callbacks e altro: Qui si creano timer ed eventuali funzioni di reindirizzamento Hardware. Ad esempio, estenderemo una callback di sistema che viene chiamata ad ogni Interrupt NVIC del timer e, mediante selezione specifica per quel timer, faremo un Toggle.
\end{itemize}

MX_GPIO_Init();

MX_I2C1_Init();
MX_TIM2_Init();
\end{verbatim}

Notiamo che ciò di cui abbiamo parlato fino ad ora viene inizializzato; tuttavia, il TIM2 (timer) non parte da solo: necessita di una funzione di start da inserire subito dopo:

\begin{verbatim}
HAL_TIM_Base_Start_IT(&htim2);
\end{verbatim}

A questo punto, ad ogni scatto viene chiamata la sua callback. Fuori dal main, la callback dovrà essere così composta:

\begin{lstlisting}[language=C]
/* USER CODE BEGIN 4 */
void HAL_TIM_PeriodElapsedCallback(TIM_HandleTypeDef *htim)
{
    if (htim->Instance == TIM2) {
        HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);  // LED blink
        printf("tick\r\n");                     // messaggio su Virtual COM
    }
}
/* USER CODE END 4 */
\end{lstlisting}

Mi aspetterò dunque un toggle ogni secondo e un printf su PC (tramite Virtual COM) che scrive `tick`. 

Si aggiunge quindi un'estensione a basso livello subito prima della callback. Normalmente, in un PC la printf() viene mandata sul terminale di default. Qui purtroppo non è cosi semplice: STM32 non ha un vero terminale virtuale, e nessun OS. Ricordando che in C (Linux): 
printf chiama la systemcall write() --> trap di sistema --> kernel --> terminale. Noi dovremo simulare lo stesso comportamento. Qui, la printf chiama \_write() che chiama \_\_io\_putchar, che andrà estesa per scrivere su UART/USART Attivato precedentemente.

% __io_putchar con USART2
\begin{lstlisting}[language=C]
int __io_putchar(int ch) {
    HAL_UART_Transmit(&huart2, (uint8_t *)&ch, 1, 10);
    return ch;
}
\end{lstlisting}

\subsubsection{Nota molto importante}

In questo primo progetto di prova le ottimizzazioni non sono l'argomento principale, si ragiona da un punto di vista puramente \textit{didattico}. Nella realtà, bisogna stare molto attenti alle operazioni di printf e, più in generale, al lato hardware.
In particolare, quando viene chiamata la callback, non bisognerebbe \textbf{mai} scrivere una printf al suo interno. questo poichè gli interrupt bloccano per definizione gli altri interrupt di stesso livello o superiore, e quindi con un'operazione piu o meno lunga (es. printf) rischia di \textit{perdersi altri interrupt} urgenti o dati in arrivo. Per questo motivo, conviene:

\begin{itemize}
    \item Evitare la stampa dentro alla callback aggiungendo solamente un aggiornamento di flag.
    \item Nel while(1) del main, stampare ogni volta che il flag cambia e poi ri-inizializzarlo.
\end{itemize}

In questo modo, la CPU si interrompe per pochissimo e, nonostante finisca i compiti di interrupt handling, rimane in ascolto.
Nel nostro caso, una soluzione possibile potrebbe quindi essere dichiarare il tick flag (sempre nella sezione 4, sopra ad \_\_io\_putchar):

\begin{verbatim}
volatile uint8_t tick_flag = 0;
\end{verbatim}

Poi cambiare la funzione di Callback:

\begin{lstlisting}[language=C]
void HAL_TIM_PeriodElapsedCallback(TIM_HandleTypeDef *htim)
{
    if (htim->Instance == TIM2) {
        HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);  // LED blink
        tick_flag = 1;                          // segnale
    }
}
\end{lstlisting}

Infine, modificare il while(1) del main per la gestione della print (o altre f):

\begin{lstlisting}[language=C]
/* USER CODE BEGIN WHILE */
while (1)
{
    /* USER CODE BEGIN 3 */
    if (tick_flag) {
        tick_flag = 0;
        printf("tick\r\n");
    }
    /* USER CODE END 3 */
}
/* USER CODE END WHILE */
\end{lstlisting}

Detto questo, resta il fatto che stiamo utilizzando un modello AI e non siamo obbligati al vincolo hard-realtime: la latenza puo essere \textit{tollerata}.

\section{Sezione 4 -- Nucleo-U545RE (MCU)}

Per motivi sperimentali, si è scelto di sospendere lo sviluppo sulla tanto usata quanto semplice F411RE per spostarsi verso un MCU Più all'avanguardia: la serie U infatti, sono boards molto più recenti: Ne è testimone la poca documentazione disponibile e, soprattutto, l'interfaccia nell'IDE (molto più rudimentale). Vediamo alcune caratteristiche chiave della
Nucleo-U545RE
\begin{itemize}
    \item Cortex-M33 a 160 MHz: più veloce rispetto ai 100MHz precedenti
    \item Flash 2MB, RAM 786KB. Entrambi 4 volte rispetto al F411RE
    \item Nuove connettività: USB C (doppio, uno anche per il caricamento diretto senza STLINK), I3C, Timer avanzati per sensori moderni, ...
    \item Consumi ridotti: Il processore è a 40nm anzichè 90, transistor quindi più piccoli e veloci con meno disspazione di energia a parità di frequenza.
\end{itemize}

Infine, la serie U5 è specificamente progettata per il consumo minimo indispensabile: oltre alle classiche run/sleep/stanby sono disponibili altre modalità di \textit{risparmio energetico}. Vi è anche una sorgente di clock Multi Speed Internal Oscillator MSIS che è specificaatamente progettata per funzionare in ultra low power.

\subsection{Nota importante}

Nota importante: l'architettura generale della board rimane la stessa: 
\begin{itemize}

\subsection{Setup iniziale}

Impostare il clock \textit{alto} giusto per prendere familiarità con lo schema delle sorgenti interne.

Partendo da sinistra, osserviamo che ci sono 2 tipi di sorgenti che ci interessano: HSI-RC e MSIS. Essi finiscono verso destra nel System Clock Multiplexer. Di default, questo Mux seleziona \textbf{MSIS}. Noi vogliamo HSI-RC (che parte da 16MHz, ma ha a disposizione divisori e moltiplicatori per aggiustarlo esattamente al nostro bisogno, a differenza di MSIS, difficilmente pilotabile).

Impostiamo quindi Mux a \textbf{PLLCLK} (arriva da HSI, da selezionare il PLL1 Source Multiplexer) e tariamo i corrispondenti M/D per arrivare a \textbf{160MHz} utilizzando semplice ragionamento \textit{logico}. Assicuriamoci infine che le uscite APB di periferiche siano anch’esse a 160MHz.

Siamo così pronti a procedere con la creazione del timer + interrupt handler, proseguendo con il nostro progetto di prova. Ovviamente, se il clock va in overflow (CLK massimo consentito 160MHz) viene mostrato un errore in viola.

\subsubsection{Conversione con quantizzazione a 8 bit}
In questo progetto di prova, prenderemo confidenza con la conversione di un modello in TensorflowLiteMicro, per poi importarlo con tutti i tool necessari per un funzionamento ottimale. Occorre precisare che, per ora, non abbiamo ancora scelto un modello da utilizzare \textit{di default}. Dunque, sarà usato un semplice modello CNN di prova sui dati campionati (No LSTM) convertito tramite \textit{TFlite Micro}.

% Script Python per quantizzazione int8
\begin{lstlisting}[language=Python]
import tensorflow as tf
import numpy as np

# 1. Carica il modello h5 (senza ricompilare, serve solo per inferenza)
model = tf.keras.models.load_model("cnn_model_no_clipnorm.h5", compile=False)

# 2. Dataset rappresentativo per calibrare la quantizzazione
def representative_dataset():
    for _ in range(100):
        # Shape corretta: (1, 30, 19) -> [batch=1, timesteps=30, features=19]
        sample_input = np.random.rand(1, 30, 19).astype(np.float32)
        yield [sample_input]

# 3. Converter configurato per int8
converter = tf.lite.TFLiteConverter.from_keras_model(model)
converter.optimizations = [tf.lite.Optimize.DEFAULT]
converter.representative_dataset = representative_dataset
converter.target_spec.supported_ops = [tf.lite.OpsSet.TFLITE_BUILTINS_INT8]
converter.inference_input_type = tf.int8
converter.inference_output_type = tf.int8

# 4. Conversione
tflite_quant_model = converter.convert()

# 5. Salvataggio su file
with open("modello_int8.tflite", "wb") as f:
    f.write(tflite_quant_model)
\end{lstlisting}

Nota: probabilmente verrà esposto un errore dovuto al training che contiene l'evaluation e altri dati. Si può \textit{ignorare} questo fatto modificando un parametro (compile=False) forzando quindi la non ricompilazione. Da notare che questo fatto può essere ignorato \textbf{se e solo se} si pensa di montare direttamente il modello per inferenza. Nel caso invece si pensa di doverlo modificare nel suo Core (es. aggiungendo/togliendo pesi in questa fase ridotta) sarà allora di vitale importanza includere il parametro per non perdersi i dati di training.

\textbf{Considerazioni finali}
\begin{itemize}
    \item Stm32cube ide non è in grado di convertire da solo ad hoc, ma solo specificando nella tendina il 'grado' di \textit{compressione} e su quale obiettivo (RAM o Flash). Puoi anche selezionare un modello h5 leggero nel caso tu sia riuscito a crearlo e runnarlo puro \textit{così com'è}.  
    \item Il modello di partenza in h5, per ora, pesa solo 1Mb. Non realistico. In ogni caso, quantizzandolo con il convertitore [senza precisione, vedi punto 3] esso risulta 1/10 (~100kB) (CNN). Invece LSTM + CNN ~1200kB --> 120kB miniaturizzato. 
    \item Quando si avrà scelto il modello, bisogna stare attenti durante la conversione: andrà scelto accuratamente (e non random, come ora) \textbf{il batch di representative dataset} su cui quantizzare: di fatti, quando il modello viene rimpicciolito, bisogna far 'notare' al modello il tipico \textbf{range} di dati di input. Così verranno 'eliminati' quegli intervalli di ingresso. Volendo fare un esempio, immaginiamo che il range della temperatura tipico sia tra i 30 e i 70 gradi. Non ha per niente senso scegliere un intervallo comprendente valori sotto zero.
    \item Quando dico 1/10 in realtà intendo \textbf{1/3}: praticamente il modello andando su proprietà dal file system vediamo che pesa 1Mb, ma in realtà la parte irremovibile in deployment sono solo i pesi (~300kB) e le attivazioni (~10kB). Il resto dell'architettura non serve su un microcontrollore e verrebbe di per sè scartata in automatico nel caso di deployment puro. Di conseguenza, l'effettiva riduzione h5 --> tflite è di 200kB \textbf{in meno} su 300kB (1/3). 
\end{itemize}

\subsubsection{Implementazione}

Quando arrivano i dati sulla \textbf{porta seriale}, viene eseguito un interrupt e l'handling viene eseguito da HAL_UART_RxCpltCallback. Dovrà essere impostata in modo da \textit{pushare i caratteri dentro a un buffer} che, se arriva a 30 righe, setta la finestra a \textit{window_ready=1}. Nota: il buffer è globale in RAM e leggibile da AI branch. 

% Callback UART RX: costruzione riga e push allo stack AI
\begin{lstlisting}[language=C]
void HAL_UART_RxCpltCallback(UART_HandleTypeDef *huart)
{
    if (huart == &huart1) {
        uint8_t c = rx_byte;

        if (c == '\n' || c == '\r' || rx_idx >= (RX_LINE_MAX - 1)) {
            rx_line[rx_idx] = 0;
            if (rx_idx > 0) {
                /* passa la riga (19 valori attesi) allo stack AI */
                ai_push_line_of_19((char *)rx_line);
            }
            rx_idx = 0;
        }
        else {
            rx_line[rx_idx++] = c;
        }

        /* riabilita ricezione 1 byte */
        HAL_UART_Receive_IT(&huart1, &rx_byte, 1);
    }
}
\end{lstlisting}

Utilizzeremo dentro questa funzione dichiarata in app_cube_AI(): 

% Push di una riga di 19 valori float nel ring buffer
\begin{lstlisting}[language=C]
int ai_push_line_of_19(const char *line)
{
    float vals[N_CH];

    if (parse_19_values(line, vals) != 0) {
        printf("ERR parse: attesi %d valori, ricevuto \"%s\"\r\n", N_CH, line);
        return -1;
    }

    /* Copia nel ring */
    if (ring_count < N_STEPS) {
        for (int i = 0; i < N_CH; i++)
            ring[ring_count][i] = vals[i];
        ring_count++;
    } else {
        /* shift (semplice): scarta la piu vecchia e aggiungi la nuova */
        for (int t = 1; t < N_STEPS; t++)
            for (int c = 0; c < N_CH; c++)
                ring[t-1][c] = ring[t][c];

        for (int i = 0; i < N_CH; i++)
            ring[N_STEPS-1][i] = vals[i];
    }

    if (ring_count >= N_STEPS) {
        window_ready = 1;
    }
    return 0;
}
\end{lstlisting}

A questo punto, acquire_process può \textit{decapsulare} il buffer globale e quantizzarlo. Poi, nel main dell'AI, il risultato dell'acquire viene dato in pasto al layer di \textbf{inferenza} ai_run (non mostrata perchè costante) e questo risultato, salvato in un altro buffer globale, verrà dato in ingresso al \textit{post_process} finale. Questo post processing si preoccuperà di de-quantizzare l'output: 

% post_process: dequantizzazione, stampa output, top-3, consumo finestra
\begin{lstlisting}[language=C]
int post_process(ai_i8 *data[])
{
    ai_i8 *src = data[0];   // 19 byte
    float f[N_CH];
    int   argmax = 0;
    float vmax   = -1e30f;

    printf("OUTPUT int8: ");
    for (int i = 0; i < N_CH; i++) {
        printf("%d%s",
               (int)((int8_t)src[i]),
               (i == N_CH-1 ? "\r\n" : ","));
    }

    printf("OUTPUT float: ");
    for (int i = 0; i < N_CH; i++) {
        f[i] = dequant_out((int8_t)src[i]);
        if (f[i] > vmax) { vmax = f[i]; argmax = i; }
        printf("%.5f%s",
               f[i],
               (i == N_CH-1 ? "\r\n" : ","));
    }

    /* top-3 */
    int   idx[3] = { -1, -1, -1 };
    float val[3] = { -1e30f, -1e30f, -1e30f };

    for (int i = 0; i < N_CH; i++) {
        float v = f[i];
        if (v > val[0]) {
            val[2] = val[1]; idx[2] = idx[1];
            val[1] = val[0]; idx[1] = idx[0];
            val[0] = v;      idx[0] = i;
        } else if (v > val[1]) {
            val[2] = val[1]; idx[2] = idx[1];
            val[1] = v;      idx[1] = i;
        } else if (v > val[2]) {
            val[2] = v;      idx[2] = i;
        }
    }

    printf("ARGMAX: %d (%.5f) | TOP3: %d:%.5f, %d:%.5f, %d:%.5f\r\n",
           argmax, f[argmax],
           idx[0], val[0],
           idx[1], val[1],
           idx[2], val[2]);

    /* Consuma la finestra: richiedi nuove 30 righe */
    window_ready = 0;
    ring_count   = 0;
    return 0;
}
\end{lstlisting}

def wait_for_ack(ser, timeout=2.0):
    """Legge dalla seriale fino a quando trova l'ACK o scade il timeout"""
    t0 = time.time()
    buf = ""
    while time.time() - t0 < timeout:
        chunk = ser.read(1024).decode(errors="ignore")
        if chunk:
            sys.stdout.write(chunk)
            sys.stdout.flush()
            buf += chunk
            if ACK_TOKEN in buf:
                return True
        else:
            time.sleep(0.01)
    return False


def main():

        ]

def create_cnn_lstm_model(window_size, n_feat, forecast_len, n_hidden=64):
    optimizer = Adam(learning_rate=0.001)
    inputs = Input(shape=(window_size, n_feat))

    x = Conv1D(64, 3, activation='relu', padding='same')(inputs)
    x = MaxPooling1D(2)(x)
    x = Dropout(0.3)(x)

    x = Conv1D(128, 3, activation='relu', padding='same')(x)
    x = MaxPooling1D(2)(x)
    x = Dropout(0.3)(x)

    # DIFFERENZA UNICA: unroll=True per eliminare TensorArray/TensorList
    x = LSTM(n_hidden, return_sequences=False, unroll=True)(x)

    x = Dense(64, activation='relu')(x)
    outputs = Dense(forecast_len * n_feat)(x)
    outputs = Reshape((forecast_len, n_feat))(outputs)

    model = Model(inputs, outputs)
    model.compile(optimizer=optimizer, loss='mse')
    return model

\subsection{Step 2 - Analisi dei dati a disposizione e opportune considerazioni}

Prima di procedere è necessario considerare il dataset di partenza. Questo passaggio è doveroso poichè si avrà, per ovvi motivi, un calo di prestazioni; se gestito nel modo corretto, l'errore può essere ridotto in un range prossimo allo 0.
Per cominciare, una buona prassi è analizzare il set di dati con cui il modello è stato allenato utilizzando dei diagrammi comuni nel mondo AI. Da notare come questa analisi è abbastanza limitata: ci servirà solo ai fini della quantizzazione. Per l'analisi approfondita, consultare il lavoro di Giacomo.
Stampiamo dunque gli istogrammi di ogni variabile, la matrice di correlazione e le loro versioni 'csv' (la vera risposta, non solo visiva).

\subsubsection{Fattori da considerare}

\begin{itemize}
    \item Campane: istogrammi a campana sono di \textit{facile gestione} (i valori stanno in un certo intorno senza scostarsi troppo dal valore medio) poichè si riescono a tagliare i valori che non si presenteranno mai. Così facendo, inserire i relativi tensori in uno spazio di indirizzamento 2$^8$ è più efficace. 
    \item Zero point: se gli istogrammi a campana non sono centrati, lo 'zero' andrà spostato \textit{verso il cuore della campana} stessa durante il processo di quantizzazione, per ovvi motivi
    \item Campane doppie: allargano i casi che si possono presentare, ma non determinano (in prima battuta) un grave problema: \textit{allargano la memoria} necessaria.
    \item Code: se, per via di glitch/guasti sono presenti anomalie nel dataset, allora sicuramente compariranno \textit{molto distanti} dalla campana dei valori normali. Queste sono le code. In quantizzazione bisogna includerle, e se questo non viene fatto, seppur in piccole quantità, si rischia il \textbf{clipping}.
    \item Clipping: 
    \begin{itemize}
        \item Il modello iniziale è allenato a vedere temperature normali tra 20° e 40°. Se arriva 60°, viene segnalato anomalo. avendo abbastanza spazio di valori potenzialmente infinito (32 o 64) è in grado di riconoscere qualsiasi valore anomalo
        \item Il modello quantizzato purtroppo non ha questo vantaggio: quando miniaturizzo devo dare un \textbf{range}, una 'vista', ovvero tutto il range di valori che potenzialmente può vedere. ANOMALIE INCLUSE, SE CI SONO STATE. Questo poichè se la 'vista' del modello quantizzato è 20°- 40°, per motivi DIGITALI i valori fuori range vengono \textbf{schiacciati} (CLIPPATI) agli estremi. 60° viene considerato esattamente come 40°. Di conseguenza, bisogna sempre includere le code nei modelli quantizzati per 'preparare' il modello a vedere eventuali anomalie, se ci sono state. In questo modo 60° sarà circa considerato un 43° (più di un regolare 40°!) In allenamento, invece, vanno escluse per \textit{ovvi motivi}.
        \item Ovviamente vanno incluse senza esagerare: questo poichè la quantizzazione funziona con l'arrotondamento al numero piu vicino. Più si allarga lo spazio, e più si perde di precisione. Ad esempio se devo rappresentare un range 0 10 in un quantizzato 0 5, e normalmente i valori oscillano tra 2 3 e 4, qualche volta 9 (anomalia), devo stare attento a non allargare. Se 2 droppa a 0q, 3 droppa a 1q, e 4 droppa a 2q $\rightarrow$ 9 è messo a 4q $\rightarrow$ OK perfetto. Se allargo troppo perchè voglio sottolineare quanto il 7 sia distante dal 4, allora si rischia: 2 droppo a 0q, 3 droppo a 0q \textbf{DI NUOVO}, ... con un bit rappresento due valori diversi. Perdo precisione laddove la campana è \textit{normale}.
    \end{itemize}
    \item Correlazione: per ovvi motivi, variabili simili ad altre per valori possono essere 'unite'.
\end{itemize}
Bisogna trovare quindi un giusto equilibrio. Dai dati, questo è possibile: abbiamo molte campane regolari, e le code sono poche e isolate. Useremo uno script per stampare i resoconti sopracitati, assieme ai csv per avere un riscontro diretto basato sui dati e non solo visivo (anche perchè individuare code in grafici non sufficientemente adattati è impossibile).

\subsubsection{Scoring}
Passaggio \textbf{fondamentale}. Si creano finestre come da modello (16 feature, 30 righe) e gliele si danno in pasto (se presente, come nel nostro caso, così è più accurato). Si vede come esso ``reagisce''. Il modello infatti ad ogni step produce normalmente una \textit{predizione} dei 10 step successivi: se questa predizione ha (tra tutti quelli di output) un valore molto alto (0 nella media, +/- 1 variazione normale, +/- 2 evento forte/raro per definizione di standardizzazione) allora significa che quella finestra è \textbf{interessante} per il modello. È successo qualcosa di fuori dalla media. Assegno come score (della finestra) il valore assoluto più alto tra tutti i valori di output (della finestra). 
Si otterrà un'insieme di finestre con diversi score. Il codice è il seguente:

\begin{lstlisting}[language=Python]
# Scoring finestre...
print("[5/6] Scoring finestre...")

predict_fn = load_savedmodel(args.savedmodel)
scores = []
n_windows_total = 0

for Xb, _ in iter_windows(
    data_std, args.window, args.forecast, args.stride, args.batch_size
):
    if predict_fn is not None:
        try:
            yb = predict_fn(Xb)
            yb = np.asarray(yb)
            score_b = np.max(np.abs(yb), axis=tuple(range(1, yb.ndim)))
        except Exception:
            score_b = np.max(np.abs(Xb), axis=(1, 2))
    else:
        score_b = np.max(np.abs(Xb), axis=(1, 2))

    scores.append(score_b)
    n_windows_total += Xb.shape[0]

if n_windows_total == 0:
    raise RuntimeError(
        "Nessuna finestra generabile: controlla window/forecast/righe del CSV."
    )

scores = np.concatenate(scores, 0)

p95 = float(np.percentile(scores, 95.0))
max_total = int(args.max_total)
k_tail = int(round(max_total * args.balance))
k_norm = max_total - k_tail

tail_mask = scores >= p95
tail_idx = np.nonzero(tail_mask)[0]

if len(tail_idx) > 0:
    tail_sorted = tail_idx[np.argsort(scores[tail_idx])[::-1]]
    tail_pick = tail_sorted[:k_tail]
else:
    tail_pick = np.array([], dtype=int)

normal_pool = np.nonzero(~tail_mask)[0]
if len(normal_pool) >= k_norm:
    rng = np.random.default_rng(args.seed)
    norm_pick = rng.choice(normal_pool, size=k_norm, replace=False)
else:
    norm_pick = normal_pool
    deficit = k_norm - len(norm_pick)
    extra_from_tail = [
        i for i in (tail_sorted[k_tail : k_tail + deficit] if len(tail_idx) > 0 else [])
    ]
    tail_pick = np.concatenate([tail_pick, np.array(extra_from_tail, dtype=int)])

selected_idx = np.unique(np.concatenate([tail_pick, norm_pick]))
if selected_idx.shape[0] > max_total:
    selected_idx = selected_idx[:max_total]

\subsection{Step 5 - Benchmarking delle prestazioni}

Per un opportuno benchmarking delle prestazioni è bene tenere a mente i parametri principali che in prima battuta ci interessano. Si noti che sono dei differenziali, indicano quindi lo scarto tra il modello normale e quello quantizzato.

\begin{itemize}
    \item MSE (Errore quadratico \textit{medio}): media degli errori tra normale e quantizzato, dando piu peso a quelli pi grossi (è quadrato)
    \item MAE (Errore medio \textit{assoluto}): simile, ma non pesa di più gli errori più forti
    \item MAX (Errore assoluto massimo): confronto tra errore massimo presente per entrambi i modelli
\end{itemize}

Un altro fattore importante è il Range dei parametri: esso indica di fatto l'intervallo di valori visibili dal modello. Se questo range nel modello INT8 è allineato con quello Keras, non si ha clipping. Viceversa, quest'ultimo sarebbe presente e alzerebbe MSE e MAX in modo non trascurabile.
Per avere un'idea dei valori ottimi, l'euristica (\textit{tratta dai paper della community}) dice:

\paragraph{Errore relativo}
\begin{equation}
E_{\mathrm{rel}} = \frac{\mathrm{MAE}}{y_{\max}^{(\mathrm{Keras})} - y_{\min}^{(\mathrm{Keras})}}
\end{equation}

Per MAE/MSE i valori ottimi sono sotto 3/5\%, mentre per MAX sotto il 10\% (buono se sotto il 20\% e RARO). Il range di Keras (denominatore) è pari a –9.88 - +3.38 = 13.26
Dopo una lunga serie di prove con n-finestre e balance diversi, la migliore configurazione possibile sembra essere con:

\begin{itemize}
    \item balance = 0.5
    \item n-finestre = 2000
\end{itemize}

Con valori di benchmarking:

\begin{itemize}
    \item MSE = 0.1399
    \item MAE = 0.2833
    \item MAX = 1.1709
    \item Range = −9.786 - 3.384 = 13.170
\end{itemize}

Il range è quasi identico, il clipping \textit{non è} quindi \textit{presente}. MSE/MAE rispettano a pieno gli scarti raccomandati. MAX può sembrare alto, ma come detto precedentemente è raro, quindi accettabile. Altre configurazioni come ad esempio 6000/0.3 oppure 4000/0.3 peggiorano (e non di poco) MSE / MAE.
Si è ora dunque pronti al deployment di questo modello.

\subsubsection{Prevedere 16 valori}

\subsubsection{Dequantizzazione Output}

Ricordando che 16 valori con una media (μ) deviazione standard (σ) calcolate sul dataset di training, la standardizzazione trasforma ogni valore grezzo x nel suo z-score x′ e ogni colonna diventa “\textit{centrata}” (media ≈ 0) e “\textit{scalata}” (dev.std ≈ 1). 

\begin{equation}
x' = \frac{x - \mu}{\sigma}
\end{equation}

Ogni valore quindi prima di essere dato in ingresso (al modello) e prima di essere considerato output vero (del modello) va quantizzato/dequantizzato a seconda di \texttt{scale} e \texttt{zero point} del tflite utilizzato. 
Questo significa che nel codice dell'STM32 i dati vanno dequantizzati in output a seconda di OUT\_Scale e OUT\_ZeroPoint del TFlite convertito utilizzato (Da leggere prima e da hardcorare nel codice). (Per ora, in ingresso sono pre-quantizzati)
Per il 2000/0.5, la coppia è la seguente: 

\begin{itemize}
    \item OUT\_Scale = 0.052055813
    \item OUT\_ZeroPoint = 62
\end{itemize}
Di conseguenza applichiamo:

\begin{lstlisting}[language=C]
for (uint32_t i = 0; i < out_len; ++i) {
    float y = ((float)q8[i] - (float)OUT_ZP) * OUT_SCALE;
    o += snprintf(msg + o, sizeof(msg) - o, "%.3f ", (double)y);
    if (o > (int)sizeof(msg) - 16) {
        log_uart(msg);
        o = snprintf(msg, sizeof(msg), " ");
    }
}
\end{lstlisting}
Ora gli output sono formattati nel modo giusto. Possiamo quindi procedere ad inviare una finestra in ingresso (STANDARDIZZATA FL32).

def fit_scaler_from_csv(csv_path: str, nfeat: int) -> StandardScaler:
    df = pd.read_csv(csv_path, header=0, sep=",", low_memory=False)
    num_df = df.select_dtypes(include=[np.number])
    if num_df.shape[1] < nfeat:
        raise RuntimeError(
            f"Calib-CSV ha solo {num_df.shape[1]} colonne numeriche, servono {nfeat}."
        )
    data = num_df.iloc[:, :nfeat].to_numpy(dtype=np.float32, copy=False)
    sc = StandardScaler().fit(data)
    return sc


def standardize(win_raw: np.ndarray, scaler: StandardScaler) -> np.ndarray:
    return scaler.transform(win_raw.astype(np.float32, copy=False)) \
                .astype(np.float32, copy=False)

\subsubsection{Scelta dei pin}
Per prima cosa, decidiamo i PIN lato MCU: TX e RX verranno mappati su PA2 e PA3 rispettivamente (Arduino D1 e D0). \subsection{Teoria e schema iniziale}

L'ultimo e importante step è quello di adottare uno standard \textit{industriale comune} (e utilizzato dal sensore di Bonfiglioli) per la lettura dei dati del sensore. Per ora, abbiamo usato un semplice file python di invio tramite ASCII. 
Partiamo dunque dal precisare che Bonfiglioli ha i due z-sensors 'slaves' collegati a un master che ne legge i valori. Noi dovremo collegare la MCU ad esso (come \textit{slave}) e sniffarne i valori, per poi procedere con la pipeline. Da notare che lo standard è binario, con il seguente schema:

\begin{verbatim}
[ADDR][FUNC][LEN][PAYLOAD][CRC_LOW][CRC_HIGH]
\end{verbatim}

\begin{itemize}
    \item 
    \item ADDR = indirizzo dello slave (1 byte)
    \item FUNC = function code (1 byte)
    \item LEN = numero di byte del payload (1 byte)
    \item PAYLOAD = dati veri e propri
    \item CRC16 = checksum finale (2 byte, little-endian)
\end{itemize}
Useremo, in ordine: 

\begin{itemize}
    \item 0x01 (arbitrario)
    \item 0x03 (Read Holding Registers)
    \item 0x040 (64 in decimale, 16 valori float (4byte ciascuno) $\rightarrow$ 64 bytes)
    \item 64 blocks scritti byte per byte in big endian (+significativo per primo)
    \item CRC L/H standard per checksum
\end{itemize}
Nota: Questo payload è custom. Ovvero, i registri modbus sono da 16 bit (2byte), non conosce i float. Quindi ogni float verrà spezzato in \textbf{due parti separate} cosi da poter essere inviato. Non sappiamo precisamente il perchè i dati sul csv siano tutti float, nonostante si utilizzi modbus RTU standard.

\begin{lstlisting}[language=Python]
def modbus_crc16(data):
    """Calcola CRC16 Modbus (polinomio 0xA001)"""
    crc = 0xFFFF
    for byte in data:
        crc ^= byte
        for _ in range(8):
            if crc & 0x0001:
                crc >>= 1
                crc ^= 0xA001
            else:
                crc >>= 1
    return crc


def build_modbus_frame(slave_addr, function_code, float_values):
    """
    Costruisce frame Modbus RTU con array di float32

    Struttura:
    [ADDR] [FUNC] [LEN] [DATA...] [CRC_LO] [CRC_HI]
    """
    # Header
    frame = bytearray()
    frame.append(slave_addr)
    frame.append(function_code)
    frame.append(len(float_values) * 4)  # Data length in bytes

    # Data: float32 big-endian
    for value in float_values:
        frame.extend(struct.pack('>f', value))  # '>f' = big-endian float

    # CRC16 (little-endian sui 2 bytes finali)
    crc = modbus_crc16(frame)
    frame.append(crc & 0xFF)         # CRC Low
    frame.append((crc >> 8) & 0xFF)  # CRC High

    return bytes(frame)
\end{lstlisting}

\subsection{Moduli di appoggio per il processing Modbus RTU}

Il file .c segue uno standard della community, ma vanno implementate ad hoc:

\begin{itemize}
    \item \texttt{modbus\_process\_byte()} : prende un byte alla volta con timestamp, riconosce la sequenza binaria decisa all'inizio e torna 1 se completo (quindi si potrà proseguire con l'estrazione del payload tramite altre funzioni)
    \item \texttt{modbus\_get\_frame()} : copia fuori il frame riconosciuto 
    \item \texttt{modbus\_extract\_float32()} : estrae il payload di 16 float
\end{itemize}

Nota: ogni funzione ovviamente non è bloccante, ma skippa blocchi successivi e torna nello stesso punto (polling) finchè non rispettano la condizione.

\subsubsection{modbus_process_byte}
\begin{lstlisting}[language=C]
bool modbus_process_byte(modbus_rx_buffer_t *buf,
                         uint8_t byte,
                         uint32_t timestamp_ms)
{
    if (!buf) return false;

    // Framing senza timing:
    switch (buf->sync_stage) {

    case 0: // Attendi indirizzo
        if (byte == MODBUS_SLAVE_ADDR) {
            buf->buffer[0] = byte;
            buf->index = 1;
            buf->sync_stage = 1;
            buf->state = MODBUS_STATE_RECEIVING;
        } else {
            // scarta
            buf->index = 0;
        }
        break;

    case 1: // Attendi function code
        if (byte == MODBUS_FUNCTION_CODE) {
            buf->buffer[1] = byte;
            buf->index = 2;
            buf->sync_stage = 2;
        } else {
            // restart sync
            buf->sync_stage = 0;
            buf->index = 0;
        }
        break;

    case 2: // Attendi LEN
        buf->buffer[2] = byte;
        buf->index = 3;
        if (byte == 0 || byte > MODBUS_MAX_DATA_LEN) {
            // len non valido -> reset
            modbus_reset(buf);
        } else {
            buf->bytes_needed = (uint16_t)byte + 2; // payload + CRC16
            buf->sync_stage = 3;
        }
        break;

    case 3: // payload+CRC
        if (buf->index < MODBUS_FRAME_MAX_SIZE) {
            buf->buffer[buf->index++] = byte;
            if (buf->bytes_needed > 0) {
                buf->bytes_needed--;
            }
            if (buf->bytes_needed == 0) {
                // Frame completo!
                buf->state = MODBUS_STATE_COMPLETE;
                buf->sync_stage = 0;
                return true;
            }
        } else {
            // overflow -> reset
            modbus_reset(buf);
        }
        break;

    default:
        modbus_reset(buf);
        break;
    }

    buf->last_rx_time = timestamp_ms;
    return false;
}
\end{lstlisting}

\subsubsection{modbus_get_frame}
\begin{lstlisting}[language=C]
bool modbus_get_frame(modbus_rx_buffer_t *buf, modbus_frame_t *frame)
{
    if (!buf || !frame || buf->state != MODBUS_STATE_COMPLETE) {
        return false;
    }

    // Parsing frame
    if (buf->index < 5) {
        modbus_reset(buf);
        return false;
    }

    frame->address  = buf->buffer[0];
    frame->function = buf->buffer[1];

    // Per Function Code 0x03 (Read Holding Registers):
    // Byte 2 = data length
    if (frame->function == 0x03) {
        frame->data_len = buf->buffer[2];

        if (frame->data_len > MODBUS_MAX_DATA_LEN ||
            buf->index < (3 + frame->data_len + 2)) {
            // Frame malformato
            modbus_reset(buf);
            return false;
        }

        // Copia dati
        memcpy(frame->data, &buf->buffer[3], frame->data_len);

        // CRC16 (ultimi 2 bytes, little-endian)
        uint16_t crc_offset = 3 + frame->data_len;
        frame->crc16_rx = (uint16_t)buf->buffer[crc_offset] |
                          ((uint16_t)buf->buffer[crc_offset + 1] << 8);

        // Calcola CRC sui dati (esclusi CRC bytes)
        frame->crc16 = modbus_crc16(buf->buffer, crc_offset);

        // Verifica CRC
        frame->valid = (frame->crc16 == frame->crc16_rx);

        // Copia raw frame
        frame->raw_len = buf->index;
        memcpy(frame->raw, buf->buffer, buf->index);

    } else {
        // Function code non supportato (per ora)
        frame->valid = false;
    }

    // Resetta buffer per prossimo frame
    modbus_reset(buf);

    return frame->valid;
}
\end{lstlisting}

\subsubsection{modbus_extract_float32}
\begin{lstlisting}[language=C]
int modbus_extract_float32(const modbus_frame_t *frame,
                           float *output,
                           int max_count)
{
    if (!frame || !output || !frame->valid) {
        return 0;
    }

    // Verifica che data len sia multiplo di 4 (float32 = 4 bytes)
    if (frame->data_len % 4 != 0) {
        return 0;
    }

    int float_count = frame->data_len / 4;
    if (float_count > max_count) {
        float_count = max_count;
    }

    for (int i = 0; i < float_count; i++) {
        output[i] = bytes_to_float32_be(&frame->data[i * 4]);
    }

    return float_count;
}
\end{lstlisting}
